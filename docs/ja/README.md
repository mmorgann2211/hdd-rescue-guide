# HDD I/Oエラー データ復旧ガイド

> 実測環境：WDC WD40PURZ-85TTDY0 4TB、1個のpending sector、Ubuntu 24.04 LTS Live USB、ddrescue 1.27、TestDisk 7.1

**Languages / 語言 / 言語:** [English](../../README.md) | [繁體中文](../zh-TW/README.md) | [日本語](README.md)

---

## 📋 使用シナリオ：まずここを確認

| 状況 | 適用可否 | 説明 |
|------|----------|------|
| HDDがI/Oデバイスエラーを表示するが、Windows/BIOSでは検出できる | ✅ 適用可 | 本ガイドの主なユースケース |
| 不良セクタが少数ある（SMART C5 > 0） | ✅ 適用可 | 早いほど成功率が高い |
| パーティションテーブルが破損（GPT damaged） | ✅ 適用可 | sgdisk --restoreで復元可能 |
| Windowsではマウントできないが、BIOSでは検出できる | ✅ 適用可 | Linuxの方がマウント耐性が高い |
| SMARTでC6（修復不可能セクタ）が大幅増加 | ⚠️ 注意 | 早急に対処、ドライブが急速に劣化する可能性あり |
| ドライブを落とした、またはカチカチ音がする | ⚠️ 注意 | 試す価値はあるが、ヘッド損傷はクリーンルームが必要 |
| PCBが焼けて完全に検出不可能 | ❌ 非対応 | ハードウェアレベルの修理が必要 |
| ヘッド損傷、起動時に持続的な異音 | ❌ 非対応 | クリーンルームでのヘッド交換が必要、費用2〜10万円以上 |

**基本原則：ドライブが検出できる限り、データを回復できる可能性がある。**

## 🚨 復旧の黄金ルール：始める前に必読

ドライブは故障の瀬戸際にある。通電のたびに、書き込みのたびに、残り少ない寿命を消費している。復旧を始める前に以下を覚えておくこと：

**1. 故障ドライブへの書き込みを直ちに停止する**  
ダウンロード、インストール、自動同期は一切不可。書き込みはデータを上書きするか、ヘッドの損傷を加速させる可能性がある。

**2. Windowsの自動修復を実行させない**  
Windowsは異常なドライブを検出するとchkdskを実行しようとする。データのバックアップ前にこれを行うと危険。修復プロンプトが表示されたらキャンセルするか、ケーブルを抜く。

**3. テストのために繰り返し電源を入れない**  
起動のたびにヘッドが再位置決めし、モーターが再起動する。損傷したドライブへの余分な物理的ストレスになる。手順を確認してから開始すること。

**4. データが安全にコピーされるまで、故障ドライブは読み取り専用として扱う**  
修復、フォーマット、パーティション操作は、データが別のドライブに安全にバックアップされるまで行わない。唯一の例外はステップ6Bのパーティションテーブル復元で、それもイメージングが完了または不可能と確認された後のみ。

**5. 時間はデータと同じ価値がある**  
不良セクタは時間とともに広がる傾向がある。問題を発見した瞬間が黄金の復旧タイミング。「後で時間があるとき」は禁物。

## 🗂️ 背景：今回の事例

このガイドは実際の復旧事例に基づいている。

**経緯：**  
約7,800時間使用したWD 4TB Purpleの監視用ドライブ（WD40PURZ）が突然「I/Oデバイスエラーのため要求を実行できません」と表示するようになった。Windowsのエクスプローラーからアクセスできなくなったが、デバイスマネージャーでは検出されていた。CrystalDiskInfoでは保留中のセクタが1個（C5 = 1）のみで、他の値は正常だった。

**結果：**  
約2〜3TBの動画・写真データが全て復旧できた。

## 🛠️ 必要な機材

### 今回実際に使用したもの

| 機材 | 用途 | 備考 |
|------|------|------|
| 故障ドライブ（WD 4TB） | 復旧元 | SATA 3.5" HDD |
| 正常ドライブ（WD 4TB） | イメージファイルと最終データの保存先 | 故障ドライブと同容量のため、イメージが切り捨てられた（FAQを参照） |
| 28GB USBフラッシュドライブ | Ubuntu Live USB起動ドライブ | システムが占有するため他の用途に使用不可 |
| 別のUSBフラッシュドライブ（容量不問） | パーティションテーブルのバックアップ | 数KBあれば十分だが、別途必要 |
| 別のコンピュータ | Live USBの作成 | 故障ドライブと同じPCは非推奨 |

### 推奨される準備（失敗を避けるために）

| 機材 | 理由 |
|------|------|
| ターゲットドライブの容量は**故障ドライブより少なくとも10%大きい**こと | 同容量だとイメージの末尾が切り捨てられ、GPTバックアップパーティションテーブルが失われる |
| 小型USBフラッシュドライブ（容量不問）を別途用意 | Live USBとは別に、パーティションテーブルのバックアップ専用として使用 |
| SATA-USB変換ケースまたは複数SATAポートのPC | 故障ドライブ、ターゲットドライブ、Live USBを同時に接続する必要がある |

## 第1部 — 状況の確認

### 方法A：Windowsで軽度の損傷（まずこれを試す）

Windowsでドライブが見えている場合、まず直接データをコピーできるか試す。

**SMARTステータスの確認：**

[CrystalDiskInfo](https://crystalmark.info/en/software/crystaldiskinfo/)をダウンロードして実行し、以下に注目：

| 属性 | 説明 |
|------|------|
| C5 代替処理保留中のセクタ数 | 0より大きいと問題セクタが存在 — I/Oエラーの主な原因 |
| C4 代替処理済みのセクタ数 | 確認済みの不良セクタ数 |
| C6 回復不可能なセクタ数 | 0より大きいと深刻な損傷 |
| 01 リードエラーレート | Seagateドライブは通常大きな値になる；WDは0に近いはず |

**シリアル番号でドライブを特定（同型番が複数ある場合）：**

```powershell
wmic diskdrive get index,model,serialnumber
```

**Robocopyでデータをコピー（軽度の損傷に適用）：**

RobocopyはWindowsの組み込みコマンド — インストール不要：

```cmd
robocopy D:\ E:\backup_D\ /E /R:1 /W:1 /LOG:C:\rescue_log.txt
```

| オプション | 説明 |
|-----------|------|
| `/E` | すべてのサブディレクトリを含む |
| `/R:1` | エラー時は1回だけリトライしてスキップ |
| `/W:1` | リトライ間隔1秒 |
| `/LOG` | 全操作を記録 — 後でERRORを検索して失敗ファイルを確認 |

完了後、メモ帳で `C:\rescue_log.txt` を開いて `ERROR` を検索し、失敗したファイルを確認する。

> ⚠️ Robocopyが最初から `エラー 1117 (0x0000045D) ソースディレクトリをスキャンしています` で失敗する場合、不良セクタがルートディレクトリまたはMFTにある。方法Bに直接進む。

> ⚠️ **データのバックアップが完了するまで、Windowsにchkdskを実行させない。** chkdskは損傷を悪化させる可能性がある。データが別のドライブに完全にコピーされた後にのみchkdskを実行する。

---

### 方法B：Linux復旧（深刻な損傷またはWindowsが完全にマウントできない場合）

Robocopyが失敗するか、再起動後にWindowsがドライブをマウントできなくなった場合は、以下に進む。

## 第2部 — Live USBの作成と起動

### 1. Live USBの作成（別のコンピュータで）

1. [Ubuntu 24.04 LTS ISO](https://ubuntu.com/download/desktop)をダウンロード
2. [Rufus](https://rufus.ie/)をダウンロード
3. USBドライブを挿入してRufusを開く
4. ISOファイルを選択
5. **⚠️ 重要：「イメージオプション」を「DDイメージモードで書き込む」に設定**（デフォルトのISOモードではない）— ISOモードでは起動できないUSBになる可能性がある
6. スタートをクリック

### 2. すべてのドライブを接続

同時に接続：
- 故障ドライブ（ソース）
- ターゲットドライブ（デスティネーション）
- バックアップ用USBフラッシュドライブ

### 3. USBから起動

BIOSでUSBを起動デバイスとして選択。Ubuntuが読み込まれたら **「Ubuntuを試す」** を選択（インストールしない）。

デスクトップに入ったらすぐにスリープを無効化：設定 → 電源 → すべて **「しない」** に設定。

## 第3部 — 復旧の実行

> ⚠️ **以下の手順の順序は非常に重要。スキップや入れ替えは禁止。**

### ステップ1：必要なツールのインストール

```bash
sudo apt update && sudo apt install gddrescue gdisk smartmontools -y
```

### ステップ2：ドライブのデバイス名を確認

```bash
lsblk -o NAME,SIZE,MODEL,SERIAL
```

シリアル番号を照合してデバイス名を確認：
- 故障ドライブ → `/dev/sdb`
- ターゲットドライブ → `/dev/sda`
- バックアップUSB → `/dev/sdd`

> ⚠️ 以下のコマンドの `/dev/sdb`、`/dev/sda` 等は実際の状況に合わせて変更すること。**デバイス名を間違えるとデータが上書きされる可能性がある。**

### ステップ3：パーティションテーブルの即時バックアップ ⭐ 最重要ステップ

> ⚠️ 他の何よりも先にこれを行うこと。後でスペース不足でイメージが切り捨てられた場合、このバックアップが命綱になる。

バックアップUSBをマウント：

```bash
sudo mkdir /mnt/usb
sudo mount /dev/sdd1 /mnt/usb
```

> USBが新品で未フォーマットの場合：`sudo mkfs.fat -F32 /dev/sdd1`、その後再度マウント。

パーティションテーブルとGPTヘッダーをバックアップ：

```bash
# 完全なパーティションテーブルのバックアップ
sudo sgdisk --backup=/mnt/usb/partition_table.bak /dev/sdb

# GPTヘッダー（ディスク先頭、プライマリパーティションテーブルを含む）
sudo dd if=/dev/sdb of=/mnt/usb/gpt_head.bin bs=1M count=1

# GPTフッター（ディスク末尾、バックアップパーティションテーブルを含む）
sudo dd if=/dev/sdb bs=512 skip=$(( $(sudo blockdev --getsz /dev/sdb) - 33 )) count=33 of=/mnt/usb/gpt_tail.bin
```

3つのファイルが存在することを確認：

```bash
ls -lh /mnt/usb
```

`partition_table.bak`、`gpt_head.bin`、`gpt_tail.bin` の3ファイルが表示されるはず — 合計2MB未満。

### ステップ4：ターゲットドライブをマウント

```bash
sudo mkdir /mnt/edata
sudo mount /dev/sda2 /mnt/edata
```

### ステップ4.5：smartctlで不良セクタの深刻度を評価

> このステップは読み取りのみ — 故障ドライブに対して完全に安全。  
> 目的：6〜15時間かけてddrescueを実行する前に、数分で不良セクタの深刻度を判断し、より大きなターゲットドライブが必要かどうかを事前に決定する。

現在のSMART値を確認：

```bash
sudo smartctl -A /dev/sdb
```

以下の3つの属性の `RAW_VALUE` 列に注目：

| 属性名 | ID | 説明 |
|--------|----|------|
| Current_Pending_Sector | C5 | 確認待ちの問題セクタ数；0より大きいとI/Oエラーの原因 |
| Offline_Uncorrectable | C6 | 確認済みの回復不可能な不良セクタ数；0より大きいと深刻な損傷 |
| Reallocated_Sector_Ct | 05 | 累積再配置セクタ数；増加し続けているとドライブが劣化中 |

SMARTショートテストを実行（約2分）：

```bash
sudo smartctl -t short /dev/sdb
sleep 120
sudo smartctl -l selftest /dev/sdb
```

**結果に基づく判断：**

| 結果 | 推奨戦略 |
|------|----------|
| C5=0、C6=0、ショートテスト合格 | ddrescueを直接実行、リスク低 → ステップ6A |
| C5が一桁、C6=0 | ddrescueを実行、ターゲットドライブに十分なスペースがあることを確認 → ステップ6A |
| C5またはC6が数十以上 | 不良セクタが多い；ddrescueは実行できるが時間がかかる。十分大きなターゲットドライブがなければステップ6Bを検討 |
| ショートテスト失敗、`Completed: read failure` が表示 | 深刻な損傷。ddrescueを実行する前により大きなターゲットドライブを入手することを推奨、そうしないとイメージ切り捨てのリスクが高い |

> 十分大きなターゲットドライブがなく、不良セクタが少数と評価される場合、ステップ6Bに直接進んで元のドライブを直接マウントしてコピーすることができる（ddrescueをスキップ）。  
> **ただし、これはイメージバックアップがないことを意味する。コピー中にドライブが完全に死亡した場合、データを復旧することはできない。自己責任でリスクを評価すること。**

### ステップ5：ddrescueでディスクイメージを作成

> ⚠️ **実行前に、ターゲットドライブの空き容量が故障ドライブの総容量を超えていることを確認：**
> ```bash
> df -h /mnt/edata
> ```
> スペースが不足している場合、**ddrescueを実行しない**。直接ステップ6Bに進む。

```bash
sudo ddrescue -d -r1 /dev/sdb /mnt/edata/rescue.img /mnt/edata/rescue.log
```

| オプション | 説明 |
|-----------|------|
| `-d` | 直接読み取り、システムキャッシュをバイパス |
| `-r1` | 不良セクタは1回だけリトライしてスキップ |

ddrescueの正常な進捗表示：
- `error rate: 0 B/s` — 不良セクタに未遭遇
- `bad areas: 0` — 損傷領域なし
- `average rate: 30〜80 MB/s` — 通常のSATA HDDの速度

所要時間：6〜15時間（ドライブ容量と不良セクタ数による）。中断された場合、同じコマンドを再実行するとrescue.logから再開できる。

ddrescueの完了後、どちらのパスに進むかを決める前にイメージの品質を確認：

```bash
cat /mnt/edata/rescue.log | grep -E "error|bad"
```

---

### ステップ6A：エラーなし → イメージファイルをマウント（より安全）

**使用条件：** ddrescueが `read errors: 0`、`bad areas: 0` で完了し、ターゲットドライブに完全なイメージのための十分なスペースがあった場合。

> ⚠️ **続行前にイメージの整合性を確認：**
> ```bash
> sudo gdisk -l /mnt/edata/rescue.img
> ```
> `GPT: damaged` が表示されるか、正しいパーティションが見つからない場合、イメージの末尾が切り捨てられている。**ステップ6Bに切り替える。**

gdiskの出力から `Basic data partition`（タイプ0700）の行を見つけ、`Start` の値（例：32768）を記録し、オフセットを計算：

```
offset = 開始セクタ × 512
例：32768 × 512 = 16777216
```

イメージをマウント：

```bash
sudo mkdir /mnt/rescue
sudo mount -o loop,ro,offset=16777216 /mnt/edata/rescue.img /mnt/rescue
ls /mnt/rescue
```

フォルダ構造が見えたら、ステップ7Aに進む。

---

### ステップ6B：エラーあり、またはスペース不足 → パーティションテーブルを復元して元のドライブを直接マウント

**使用条件：** ddrescueに `read errors` または `bad areas` がある、またはターゲットドライブのスペース不足でイメージが切り捨てられ、イメージをマウントできない場合。

> ⚠️ **このステップは故障ドライブに直接書き込む。** パーティションテーブルのみを変更しデータには触れないが、損傷を悪化させる極めて低いリスクがある。  
> 前提条件：ステップ3のパーティションテーブルバックアップが完了していること。  
> ステップ3が完了していない場合、**先にステップ3を実行すること。**

バックアップからGPTを復元：

```bash
sudo sgdisk --restore=/mnt/usb/partition_table.bak /dev/sdb
```

読み取り専用モードでデータパーティションをマウント：

```bash
sudo mkdir /mnt/rescue
sudo mount -o ro /dev/sdb2 /mnt/rescue
ls /mnt/rescue
```

フォルダ構造が見えたら、ステップ7Bに進む。

---

### ステップ7A：イメージからターゲットドライブにデータをコピー

> ⚠️ **ターゲットドライブにイメージファイルと抽出データの両方に十分なスペースがあることを確認：**
> ```bash
> df -h /mnt/edata
> du -sh /mnt/rescue
> ```
> スペースが不足している場合、先にイメージを削除してからコピーする（ステップ7Bの削除手順を参照）。

```bash
sudo cp -av /mnt/rescue/* /mnt/edata/
```

`-av` でコピー中の各ファイルが表示される。完了するとターミナルが `ubuntu@ubuntu:~$` プロンプトに戻る。

---

### ステップ7B：元のドライブからターゲットドライブにデータをコピー

> ⚠️ **rescue.imgを削除する前に、ステップ3のパーティションテーブルバックアップが完了していることを確認すること。** イメージは削除後に復元できないが、USB上のパーティションテーブルバックアップは引き続き使用できる。

イメージを削除してスペースを解放：

```bash
sudo rm /mnt/edata/rescue.img
```

スペースが解放されたことを確認：

```bash
df -h /mnt/edata
```

元のドライブからターゲットにデータをコピー：

```bash
sudo cp -av /mnt/rescue/* /mnt/edata/
```

完了するとターミナルが `ubuntu@ubuntu:~$` プロンプトに戻る。

---

## 第4部 — 完了と確認

### 復旧結果の確認

```bash
df -h /mnt/edata
ls /mnt/edata
```

容量が適切に増加し、フォルダが存在することを確認する。

### Windowsに戻した後

- ターゲット（正常な）ドライブはそのまま接続して読み取り可能 — 追加の手順は不要
- 故障ドライブを再接続した後、WindowsがchkdskをHDDに実行するよう求めた場合、**今は安全に実行させてよい** — データは安全にバックアップ済み
- 故障ドライブは引退させること。主要なストレージとして使用しない。

## 付録：よくある質問

**Q：Rufusでどのモードを使えばよい？**  
A：「DDイメージモードで書き込む」を選択。デフォルトのISOモードでは起動できないLive USBになる場合がある。

**Q：Robocopyがスキャン中に直ちにI/Oエラーで失敗する？**  
A：不良セクタがルートディレクトリまたはMFTにある。Windowsツールではこれを回避できないため、Linux ddrescueを使用すること。

**Q：ddrescueが非常に遅く、毎秒数KBしか出ない？**  
A：不良セクタのリトライ中は正常。`-r1` でセクタごとのリトライを1回に制限してスキップする。全体的な進捗が続いている限り、完了する。

**Q：`NTFS signature is missing` でイメージのマウントが失敗する？**  
A：イメージが切り捨てられ（ターゲットドライブのスペース不足）、GPTバックアップパーティションテーブルが失われた可能性が高い。ステップ3が完了している場合、`sgdisk --restore` で元のドライブのパーティションテーブルを復元して直接マウントする。

**Q：TestDiskが多くの奇妙なパーティションを見つけ、自分のデータが見つからない？**  
A：GPTが破損している場合、TestDiskのスキャン結果には多くの誤検知が含まれる。パーティションテーブルバックアップから復元する方がTestDiskのスキャンより信頼性が高い。

**Q：ターゲットドライブが故障ドライブと同じサイズでも大丈夫？**  
A：技術的には可能だが、イメージの末尾数百MBが切り捨てられ、GPTバックアップパーティションテーブルが失われてマウントに失敗する。対処法：ステップ3でパーティションテーブルをバックアップしてから、`sgdisk --restore` で元のドライブに復元する。

**Q：復旧中にコンピュータがスリープした場合はどうする？**  
A：ddrescueとcpの両方が中断される。開始前に設定 → 電源 → すべて「しない」に設定すること。ddrescueは同じコマンドを再実行することで再開できる。cpは最初からやり直す必要がある。
